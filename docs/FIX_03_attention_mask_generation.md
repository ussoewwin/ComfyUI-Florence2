# Fix #3: Missing attention_mask in generate()

## Root Cause

`Florence2ForConditionalGeneration.generate()` was not passing the `attention_mask` to
the internal `language_model.generate()` after merging image features with text inputs.

This is a **latent bug in the original code**, independent of the transformers version.
However, in v4.x, `GenerationMixin.generate()` had internal logic that inferred an
`attention_mask` from `inputs_embeds.shape` when `attention_mask=None` was received
(especially when `num_beams=1`), so it happened to work by coincidence.

In v5.x, the inference logic inside `GenerationMixin.generate()` was changed,
and `attention_mask=None` is no longer properly compensated for.
The result is **no errors, but empty output** -- the most insidious type of failure.

This is the **direct cause** of "the developer says errors are fixed, but the functionality is dead."

---

## Technical Details

### Florence-2 generate() Flow

```
Florence2ForConditionalGeneration.generate()
    |
    +- 1. input_ids -> inputs_embeds conversion
    +- 2. pixel_values -> image_features via _encode_image()
    +- 3. Merge [image_features, text_embeds] -> inputs_embeds
    |     Also generates attention_mask for the merged sequence
    |
    +- 4. language_model.generate(inputs_embeds=..., attention_mask=???)
                                                     ^^^^^^^^^^^^^^^^
                                                     This is the problem
```

In step 3, `_merge_input_ids_with_image_features()` returns both `inputs_embeds` and `attention_mask`.
However, **in step 4, the `attention_mask` was not passed to `language_model.generate()`**.

### `_merge_input_ids_with_image_features()` Implementation

```python
# modeling_florence2.py L2664-2686
def _merge_input_ids_with_image_features(self, image_features, inputs_embeds):
    batch_size, image_token_length = image_features.size()[:-1]
    device = image_features.device
    image_attention_mask = torch.ones(batch_size, image_token_length, device=device)

    task_prefix_embeds = inputs_embeds
    task_prefix_attention_mask = torch.ones(batch_size, task_prefix_embeds.size(1), device=device)

    # concat [image embeds, task prefix embeds]
    inputs_embeds = torch.cat([image_features, task_prefix_embeds], dim=1)
    attention_mask = torch.cat([image_attention_mask, task_prefix_attention_mask], dim=1)

    return inputs_embeds, attention_mask
```

This function correctly generates and returns the `attention_mask`.
The problem was that the caller ignored it.

## Code Details

### Before (Original)

```python
def generate(self, input_ids, inputs_embeds=None, pixel_values=None, **kwargs):
    if inputs_embeds is None:
        if input_ids is not None:
            inputs_embeds = self.get_input_embeddings()(input_ids)
        if pixel_values is not None:
            image_features = self._encode_image(pixel_values)
            inputs_embeds, attention_mask = self._merge_input_ids_with_image_features(image_features, inputs_embeds)
            # ^ attention_mask is received but...

    return self.language_model.generate(
        input_ids=None,
        inputs_embeds=inputs_embeds,
        # v attention_mask is NOT passed!
        **kwargs
    )
```

`attention_mask` is set as a local variable, but it is not included in `**kwargs`
(it was neither popped from `kwargs` nor explicitly passed).

### After (L2798-2823)

```python
def generate(self, input_ids, inputs_embeds=None, pixel_values=None, **kwargs):

    attention_mask = kwargs.pop('attention_mask', None)

    if inputs_embeds is None:
        if input_ids is not None:
            inputs_embeds = self.get_input_embeddings()(input_ids)
        if pixel_values is not None:
            image_features = self._encode_image(pixel_values)
            inputs_embeds, attention_mask = self._merge_input_ids_with_image_features(image_features, inputs_embeds)

    if attention_mask is None and inputs_embeds is not None:
        attention_mask = torch.ones(inputs_embeds.shape[:2], device=inputs_embeds.device, dtype=torch.long)

    return self.language_model.generate(
        input_ids=None,
        inputs_embeds=inputs_embeds,
        attention_mask=attention_mask,
        **kwargs
    )
```

## Rationale

### 1. `kwargs.pop('attention_mask', None)`

If an `attention_mask` is passed externally (e.g., generated by ComfyUI's processor),
extract it from `kwargs` for use.
Using `pop` prevents duplication when `**kwargs` is passed through again.

### 2. Override by `_merge_input_ids_with_image_features()`

When image input is present, the `attention_mask` generated by this function
has the correct length for image tokens + text tokens.
It takes precedence over externally provided masks (which only cover text).

### 3. Fallback Generation

For cases with no image input and no external mask,
an all-ones `attention_mask` is generated from the `inputs_embeds` shape.

### 4. Explicit Forwarding to `language_model.generate()`

The most critical fix. Explicitly passing `attention_mask` to `language_model.generate()`
allows transformers' `GenerationMixin` to apply the correct `attention_mask` to the encoder.

---

## Why attention_mask Matters

### Role of attention_mask in Encoder-Decoder Models

Florence-2's text generation component is a BART-based Encoder-Decoder model.

```
inputs_embeds = [image tokens (577)] + [text tokens (variable)]
                 |
            Encoder (attention_mask specifies valid tokens)
                 |
            Decoder (generation loop)
```

When `attention_mask` is `None`, the Encoder cannot determine which tokens to attend to.

### Severity with Beam Search

With `num_beams > 1` (beam search), the problem becomes far more severe.
Beam search replicates the input for each beam, but with `attention_mask=None`,
the correspondence between replicated inputs and actual content breaks down. This results in:

- Empty string output
- Output of only `</s>` (immediate termination)
- Nonsensical string output

## Related Fix: `num_beams` Default Value Change

### Modified File

`nodes.py`

### Changes (L379)

```python
"num_beams": ("INT", {"default": 1, "min": 1, "max": 64}),
```

The original default was `3`.

### Reasons for the Change

1. **Beam search issue mitigation**: While the `attention_mask` fix enables beam search to work correctly,
   defaulting to greedy decoding (`num_beams=1`) reduces the risk of edge-case issues.

2. **Inference speed**: Beam search requires parallel processing proportional to the beam count,
   increasing VRAM usage and inference time. For most use cases, `num_beams=1` is sufficient.

3. **Consistency with `do_sample=True`**: The original code had both `do_sample=True` (sampling) and
   `num_beams=3` (beam search) as defaults simultaneously.
   While this combination is valid, it is prone to unexpected behavior.

---

## Why This Bug Was Hard to Detect

1. **No error is raised**: `attention_mask=None` is a valid input, and transformers handles it
   internally without exceptions or warnings in the log.

2. **It happened to work in v4.x**: The v4.x `GenerationMixin` had logic to auto-generate
   an all-ones mask from `inputs_embeds.shape` when `attention_mask=None`.
   This inference logic was changed in v5.x.

3. **Developers focused on "error fixes"**: Explicit errors from the transformers v5.x transition
   (`ImportError`, `ValueError`, etc.) were fixed, but the functional bug of
   "no errors but empty results" was overlooked.

4. **Symptoms are environment-dependent**: With `num_beams=1` and certain model sizes,
   partial functionality may be observed. The issue is especially pronounced with beam search.

## Test Environment

- transformers 5.4.0
- Tested with num_beams=1 (greedy) and num_beams=3 (beam search)
- Verified across caption, detailed_caption, OD, OCR and other tasks

## Backward Compatibility

Explicitly passing `attention_mask` is correct behavior in transformers v4.x as well, with no side effects.
In fact, it should have been explicitly passed in v4.x too (rather than relying on implicit inference).
